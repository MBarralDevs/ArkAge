# Plan E — v1.5 Reframing: Circle Agent Stack Integration

> **For agentic workers:** This is a **strategic plan**, not a task-by-task execution plan. Themes are coarser-grained than Plans A-D. Detailed task-by-task breakdowns live in per-theme follow-up plans (`plan-e1-*.md`, `plan-e2-*.md`, …) that get written when each theme is greenlit for execution. Use `superpowers:brainstorming` before turning a theme into an execution plan.

**Context:** Circle launched the **Agent Stack** on 2026-05-09 — five products that overlap meaningfully with assumptions we made in the v1 spec (specs/2026-05-02-arkage-design.md):
- **Agent CLI** — `circle init` / `circle agent run` scaffolding
- **Agent Wallets** — managed wallet provisioning purpose-built for AI agents (programmable, policy-aware, custodial w/ user-controlled keys)
- **Nanopayments** — Circle's productized x402 batched scheme (the SDK we already wrap)
- **Marketplace** — `agents.circle.com`, a discovery surface for agents that monetize via Nanopayments
- **Skills** — pre-built capability bundles agents can compose at runtime

**Goal:** Selectively adopt the parts of the Agent Stack that make ArkAge **easier to onboard, easier to operate, and more visible** — without surrendering the parts that make ArkAge **a protocol** rather than a Circle SDK wrapper. Drop the workarounds Circle now obsoletes. Sharpen the differentiators Circle does not address.

**The strategic reframing in one sentence:**

> ArkAge sits **one layer above** Circle's plumbing — the on-chain coordination protocol for agentic commerce (jobs, identity, reputation, disputes, programmable hooks, permissionless registry). Circle Agent Stack is excellent infrastructure to plug into; it is not the protocol layer.

This frames every theme below: where Circle gets stronger, we lean on it; where Circle is silent (programmable settlement, cross-evaluator reputation, dispute UX, permissionless registration), we double down.

**Tech Stack additions (corrected 2026-05-11 after reading Circle docs):**
- `@circle-fin/cli` — global npm CLI (`circle wallet ...`, `circle services pay`, `circle gateway ...`). **There is no programmatic SDK for Agent Wallets** — all ops are CLI subprocess invocations.
- ERC-8004 IdentityRegistry + ReputationRegistry on-chain registration (long-promised; v1 punted)
- `agents.circle.com/services` listing (Circle marketplace — discovery only, no allowlist)
- **Dropped from initial Tech Stack:** ERC-7710 session keys via Circle. **Not currently shipped by Circle Agent Wallets.** See updated Theme C.

**Plan reference:** Builds on Plans A-D (v1, all complete and tagged). Spec extensions captured in this document; promote into a `specs/2026-05-11-arkage-v1.5-design.md` once Themes A-D land. **Theme A execution detail in `plan-e1-circle-agent-wallets-onramp.md`.**

## Reality-check addendum (2026-05-11)

After reading Circle's official docs (memory: `reference_circle_agent_wallets_facts.md` + `reference_circle_agent_stack_docs.md`), four assumptions in the original Plan E themes needed correction:

| Original Plan E claim | Corrected understanding |
|---|---|
| "Circle Agent Wallets ship with a programmable SDK" | **CLI-only.** `@circle-fin/cli` (Node 20.18.2+). Auth = email OTP, 7-day sessions. No API key mode. |
| "ArkAge provisions Agent Wallets server-side via `bootstrap_user`" | **Impossible** — sessions are bound to a builder's email inbox. ArkAge can only **accept** Circle-managed addresses as Tier 2 inputs. |
| "Mirror Circle's policy schema into PolicyHook" | **Custom policies are mainnet-only.** On Arc Testnet (where ArkAge lives), PolicyHook stays the sole policy layer — already the v1 design, so no rework needed. |
| "ERC-7710 session keys via Circle Agent Wallets policy engine" | **Not shipped.** Custom policies are only allow/blocklists + spend caps. Theme C is **on ice** until Circle delivers session keys. |

What we **gain** from the Agent Stack that v1 lacked:
- Auto-funded testnet wallets (20 USDC on first login, no faucet bookmark needed)
- Gas sponsorship for all Agent-Wallet flows (Tier 3 gas-funder becomes optional for Circle-backed paths)
- Sanctions screening built in
- `circle services pay <url>` — verified compatible with ArkAge's `arkage-proxy` (same protocol, same SDK underneath, no allowlist). The v1 smoke already proved the SDK path; the CLI is a wrapper over it.

Strategic positioning is **strengthened**, not weakened: ArkAge structurally cannot custody-leak Circle wallets — Circle's design enforces it. ArkAge is unambiguously the on-chain coordination + reputation + dispute layer above Circle's wallet plumbing.

## What we keep from v1 (do not regress)

These are load-bearing. The Agent Stack does **not** replace them:

1. **HookComposer + PolicyHook + ReputationHook + EvaluatorFeeHook** — programmable settlement hooks plugged into ERC-8183. Circle's Nanopayments has no equivalent.
2. **`x402PaymentSession` workflow with reputation gates** — every 10 receipts, ArkAge checks ERC-8004 reputation and can short-circuit the session. Circle's facilitator is reputation-agnostic.
3. **`x402DisputeFlow` workflow** — opens disputes from receipts, pulls facilitator logs, settles refunds. Circle has no dispute primitive for x402.
4. **Reason-field threading** — the `bytes32 keccak256(canonicalize(evaluatorOutput))` link between off-chain evidence and on-chain settlement + reputation. Anyone-can-verify via `arkage:verify_evidence`.
5. **Permissionless AgentRegistry** — any builder can register an agent. `agents.circle.com` will be a curated marketplace; we are the open ledger underneath.
6. **MCP-first tool surface** — 28 tools. Circle's Skills system is complementary (capabilities) rather than competing (state + commerce).
7. **Tier-based wallet routing** — Tier 1 / Tier 2 / Tier 3 separation. Circle Agent Wallets slot **into** Tier 2 (see Theme A); they do not eliminate the topology.

## What we drop

Workarounds we built specifically because Circle had not yet shipped agent-native primitives. The Agent Stack now obsoletes them.

| Dropped | Why it existed | What replaces it |
|---|---|---|
| `ARKAGE_TIER2_KEY_<walletId>` env-staged private keys | LBC-1: Circle DCW EOA-mode lacked policy-aware programmable signing for agents | Circle Agent Wallets with on-wallet policy engine + ERC-7710 session keys (Theme C) |
| `smoke-register-external-tier2.ts` workaround for testnet-funded MetaMask EOAs | No agent-friendly wallet provisioning UX existed | `bootstrap_user` accepts Circle-provisioned agent wallets natively (Theme A) |
| Manual MCP token issuance scripts (`smoke-issue-token.ts`) for non-passkey testing | No Circle-side identity binding | Circle Agent Wallets carry a `circle-agent-id` that we bind to ArkAge builder identity (Theme A) |
| `pay_and_call` URL-parsing endpoint-id workaround | x402 receipts had no canonical endpoint ID exposed by the SDK | Re-evaluate if Circle's productized Nanopayments exposes a receipt-side endpoint reference (Theme D research, may stay) |

## Theme A — Onramp simplification via Circle Agent Wallets

**Problem we are solving:** v1's onramp stages a Tier 2 private key as `ARKAGE_TIER2_KEY_<walletId>` env var, which violates "no plaintext keys at rest" hygiene we tolerate only on testnet. We want a clean replacement.

**Corrected reframe (post-docs):** Builders run `circle wallet login` on their own machine. Circle auto-provisions an Agent Wallet on Arc Testnet (chain code `ARC-TESTNET`) with 20 USDC pre-funded. The builder pastes the wallet address into ArkAge's console; ArkAge registers it with `kind: "circle-agent-wallet"`. The agent runtime — wherever it lives (builder's machine, container, VM) — uses the local Circle CLI session to act.

ArkAge **does not hold the Circle session.** Structurally cannot, by Circle's design. This is a feature, not a limitation: it eliminates the entire class of "ArkAge custody compromise" risk.

**Outcomes:**
- No private keys staged anywhere in ArkAge's infra.
- 20 USDC auto-fund replaces manual faucet trips.
- Gas-sponsored on-chain ops via Circle for any flow through the Agent Wallet.
- Sanctions screening on every transfer (Circle does it).
- `circle services pay` works against ArkAge `arkage-proxy` URLs out of the box (protocol verified).

**In scope (Plan E1):**
- Migration: add `kind` enum + `circle_agent_wallet_email` column to `wallets` table.
- `src/lib/circle-cli.ts` — subprocess wrapper around `circle` for **smoke scripts + local agent runtimes only** (NOT Vercel functions — no CLI in serverless).
- Extend `arkage:register_agent_wallet` MCP tool to accept `kind: "circle-agent-wallet"`.
- Extend `arkage:bootstrap_user` for the Circle path.
- Route `arkage:pay_and_call` based on Tier 2 kind: Circle-backed wallets return an envelope pointing the builder at `circle services pay`; other kinds keep v1 behavior.
- Console UI: "Use Circle Agent Wallet" tab with copy-paste CLI commands.
- New smoke `scripts/smoke-register-circle-agent.ts` (replaces `smoke-register-external-tier2.ts`).

**Out of scope:**
- Mainnet rollout (Theme F).
- Re-implementing Tier 1 (Tier 1 stays as builder-level Circle Modular Wallet w/ passkey).
- Programmatic Agent Wallet provisioning (Circle hasn't shipped it).
- Policy mirroring (custom policies are mainnet-only at Circle; testnet has none).

**Risks:**
- `circle services pay` JSON response shape isn't fully documented — Plan E1 pre-flight captures real examples before code lands.
- Builder UX assumes the builder is comfortable with a CLI. Lower-friction "browser-only" onramp would require Circle to ship a hosted login → token exchange flow; not available today.

**Execution plan:** `plan-e1-circle-agent-wallets-onramp.md` (drafted, ready to execute).

## Theme B — On-chain ERC-8004 + AgentRegistry registration

**Problem we are solving:** v1 ships agents that exist **only in Postgres**, not on-chain. The spec promised both, but Plan B deferred on-chain registration ("Option B" of the smoke playbook) because Tier 2 EOAs lacked a clean signing path. The user explicitly asked that Option B land in v1.5.

**Reframe:** Every registered agent gets:
1. An ERC-8004 IdentityRegistry NFT (owned by **Tier 1**, per the spec — Agent Wallets do NOT own NFTs)
2. A `ReputationRegistry` initial-feedback entry (zero score, agent's `feedbackHash` slot allocated)
3. An entry in our `AgentRegistry` contract (links agent → operator wallet → policy URI)

These are three separate calls; bundled behind a single `arkage:register_agent_onchain` MCP tool. The existing `register_agent` (off-chain only) becomes a draft mode; on-chain registration is the "publish" step.

**Outcomes:**
- Agents are discoverable on-chain by anyone, not just via ArkAge's API.
- Reputation is portable: any third party can read `ReputationRegistry.getFeedback(agentId)` without going through ArkAge.
- We earn the "permissionless registry" positioning we claim in marketing.
- Foundational requirement for marketplace listings (Theme D) — `agents.circle.com` will almost certainly want an on-chain anchor.

**In scope:**
- New MCP tool `arkage:register_agent_onchain` (Tier 1 signature required).
- Workflow extension to `bootstrap_user`: post-Postgres-registration, optionally enqueue the three on-chain registrations.
- Indexer additions to Goldsky pipeline for new event topics (IdentityRegistered, FeedbackHashUpdated, AgentRegistered).
- Dashboard surface: agent profile pages show on-chain anchor (NFT token ID, registry tx hash) when present, "draft" badge when not.
- `arkage:verify_evidence` extended to cross-check the on-chain `feedbackHash` against off-chain evidence blob.

**Out of scope:**
- Migrating existing v1-registered agents to on-chain (one-time batch op; runbook, not protocol).
- Cross-chain registry (mainnet only, Theme F).

**Risks:**
- Gas cost on Arc Testnet is negligible; on mainnet (eventually) it is not. Need to batch via Multicall3 or accept that mainnet registration is a billable action.
- The four ERC-8004 / ERC-8183 testnet addresses pinned in CLAUDE.md are tutorial-sourced. **Mandatory: re-verify via `cast code` before Theme B implementation starts.** If redeployed, update spec.

**Execution plan link:** to be written as `plan-e2-on-chain-registration.md`.

## Theme C — ERC-7710 session keys (ON ICE)

**Status (2026-05-11):** **Deferred.** Circle Agent Wallets do not currently ship ERC-7710 session keys. Their custom-policy DSL covers spend caps + allow/blocklists only; no allowed-selectors, no signing-scope restrictions, no time-windowed delegation primitive.

**Why this is OK:**
- Theme A's "ArkAge does not hold the Circle session" posture already removes the most acute v1 risk (`ARKAGE_TIER2_KEY` env-staged plaintext keys at rest on Vercel). The session lives on the builder's machine; Circle's MPC architecture protects the actual key shares.
- The remaining risk surface (an agent runtime over-signing on the builder's own machine) is no worse than any other locally-run wallet, and is structurally limited by Circle's testnet auto-fund cap.

**When to revisit:**
- Circle Agent Wallets ship ERC-7710 or an equivalent session-key DSL (watch their changelog).
- Or: ArkAge needs to support a deployment model where the agent runtime is hosted by ArkAge (not the builder) — at which point we re-acquire session-control risk and need scoped delegation.

**If/when reactivated:** the original sketch is preserved in git history; re-read and revise against whatever Circle ships.

## Theme D — Distribution via agents.circle.com

**Problem we are solving:** ArkAge has zero discovery surface today. The dashboard shows the agent economy, but nobody knows the dashboard exists. Circle's marketplace is the obvious distribution channel for agents that monetize via Nanopayments — which is exactly our agents.

**Reframe:** Every agent registered via ArkAge can opt into a marketplace listing. The listing carries:
- Agent name, description, capabilities (from `register_agent` metadata)
- x402 endpoints (price per call, hosting mode, sample request)
- On-chain reputation badge (cross-linked to ERC-8004 anchor from Theme B)
- Link back to ArkAge agent profile page (for dispute history, evaluator outputs, evidence trail)

The marketplace becomes the **shop window**; ArkAge remains the **back of house** (registry, reputation, disputes, programmable settlement).

**Outcomes:**
- Distribution: agents listed on `agents.circle.com` get organic traffic from Circle's ecosystem.
- Trust transfer: agents that have transacted via ArkAge carry reputation badges, differentiated from raw Circle-only listings.
- Reverse funnel: marketplace visitors click through to `arkage.network/agents/<id>` to inspect evidence; they discover ArkAge as the trust layer.

**In scope:**
- New MCP tool `arkage:publish_to_marketplace` (Tier 1 signature, idempotent).
- New `src/lib/circle-marketplace.ts` — API client.
- Dashboard surface: "Publish to Circle marketplace" CTA on builder console agent pages.
- One-way sync cron: marketplace listings updated when reputation deltas exceed a threshold.

**Out of scope:**
- Reverse sync (marketplace → ArkAge). The marketplace is a derived view, not the source of truth.
- Listing-fee monetization (no, listings are free for v1.5 — ArkAge monetizes via evaluator fees and treasury reconciliation, not registry tax).

**Risks:**
- Marketplace API likely has rate limits and content moderation. Plan for retries + graceful 403 handling.
- If `agents.circle.com` introduces its own reputation system, we MUST cross-link to ERC-8004 anchors to avoid reputation fragmentation. Lobby Circle for cross-link support if not present at launch.

**Execution plan link:** to be written as `plan-e4-marketplace-distribution.md`. **Depends on Theme B (on-chain anchor for badge).**

## Theme E — Sharpen differentiators

**Problem we are solving:** As Circle's stack matures, the surface where ArkAge is "Circle plus a database" gets bigger. We need to invest in the surfaces where ArkAge is **structurally** ahead — programmable settlement, dispute resolution, cross-evaluator reputation. These were planned for v1.5 anyway; v1.5's strategic positioning makes them urgent.

**Sub-themes:**

**E.1 — Dispute UX overhaul.** `x402DisputeFlow` exists but is bare. v1.5 ships:
- A dispute timeline view in the dashboard with evidence inline.
- Email/webhook notifications on dispute state transitions.
- Pre-built dispute templates for common scenarios (no response, malformed response, wrong content type).
- Counter-party response window with auto-escalation.

**E.2 — Reputation UX.** v1 stores reputation; v1.5 surfaces it usefully:
- Agent profile pages show reputation distribution histogram (not just a number).
- Evaluator-level filtering ("rep among Sonnet evaluators vs Haiku evaluators").
- Time-decay badge ("most reviews in last 30 days" vs "no activity since X").
- Cross-counterparty diversity score (rep from 1 buyer ≠ rep from 50 buyers).

**E.3 — Programmable hook showcase.** Today `PolicyHook` is the only non-trivial hook in production use. v1.5 ships two more reference hooks to prove the composer pattern:
- `RoyaltyHook` — sends N% of every settlement to a creator address. Useful for agent-as-IP scenarios.
- `RateLimitHook` — on-chain rate limiting independent of policy. Caps cross-builder spam attacks.

**E.4 — Evaluator marketplace.** The BYO-evaluator path exists in v1 but is hidden. v1.5 surfaces it:
- A list of public evaluators (Claude tiers + community-registered).
- Per-evaluator reputation (which evaluators produce upheld dispute outcomes vs overturned ones).
- Builders can pick an evaluator per job-class in the console.

**In scope:** all four sub-themes. They are individually small but cumulatively essential.

**Out of scope:** anything that needs new on-chain contracts beyond `RoyaltyHook` + `RateLimitHook` (defer to v2).

**Execution plan link:** four small follow-up plans, `plan-e5-{disputes,reputation,hooks,evaluators}.md`. Some can ship in parallel.

## Theme F — Mainnet readiness

**Problem we are solving:** Arc mainnet is on the roadmap. ArkAge v1 is testnet-only. v1.5 does the prep work so v2 can flip a switch.

**In scope (v1.5):**
- Network parameterization audit: every chain-id, every facilitator URL, every contract address must come from env, not be hardcoded. Spot-check Plans A-D for missed cases.
- Mainnet deployment script for the 5 ArkAge contracts (Foundry script — deploy-only, no broadcast).
- Address registry: `contracts/deployments/arc-mainnet.json` placeholder file.
- Cost modeling: estimate gas costs for `register_agent_onchain` on mainnet. If >$5 per registration, add Multicall3 batching path before going live.
- Secrets rotation runbook: which Vercel envs need separate prod vs testnet values.

**Out of scope (defer to v2):**
- Actual mainnet deploy (waits for Arc mainnet GA + ArkAge legal review).
- Cross-chain (Base, Optimism). Single-chain v1.5; multi-chain v2.

**Execution plan link:** `plan-e6-mainnet-prep.md`. Mostly audit + scripts.
